MultipartFile 처리 문제
문제 인식
Spring Security를 추가한 뒤, 대용량 파일을 업로드할 때마다 서버에서 500 Internal Server Error가 발생했습니다. 이로 인해 프론트엔드 팀원이 jwt 인증과정에서 생긴 문제인지, 파일업로드 관련 문제인지 구별하기 어려웠습니다. 기존에 잘 사용하던 @ExceptionHandler에서 발생할 수 있는 파일 업로드 관련 모든 Exception을 쪼개봐도 커스텀 예외 처리가 안된다는 점을 발견하였습니다.
문제 해결
20MB의 용량이 넘는 파일 하나를 요청해보고, 이력서 이미지와 파일을 합쳐 30MB 크기를 전송해 비교한 결과, 두개의 에러 로그가 다른 점을 발견했습니다.
요청 크기 제한을 초과했을 때는 SizeLimitExceededException이 발생하였고, 이 예외는 MVC 패턴에 넘어오기전 톰캣에서 발생했다는 것을 알 수 있었습니다.
따라서 SizeLimitExceededException을 ExceptionHandler에서 처리하기 위해 application.properties에서 server.tomcat.max-swallow-size 속성을 설정하였습니다. 이를 통해 예외가 잡히는 것을 확인하였고, @ResponseStatus(HttpStatus.PAYLOAD_TOO_LARGE)를 추가하여 예외 처리를 진행하였습니다.
Security 인증/인가 문제
문제 인식
주요 MVP 개발 이후 2차 개발 과정에서 Security를 적용하자 Authentication 과정에서 NumberFormatException 문제가 발생하였습니다.
원래대로라면, 회원로그인 → AuthService → UserDetailsService → JwtFilter Authorization → AccessToken, RefreshToken 응답 과정을 거쳐 Authentication/Authorization 과정이 이루어지는데 UserDetailsService에서 NumberFormatException이 발생하였습니다.
문제 해결
문제가 User Class에서 일어났으니, 해당 부분에 중단점을 찍어 디버깅을 진행했습니다.
Authentication 이 진행될 때, User Class에 들어가는 username 부분 값을 확인해보니, memberId가 아니라 email이 들어가는 것을 알 수 있었습니다.
따라서 초기에는 memberId로 Authentication을 구현하는 것이 의도였으나, 차후 인증이 필요한 API 호출에서 로그인한 회원의 정보를 쓰는 경우가 많을 것을 예상해 CustomUser Class 필요성을 인지하였습니다.
이에따라 다른 클래스에서도 인증된 사용자에 대한 email, memberId, role, social 정보가 필요하기 때문에, CustomUser를 만들어 사용해 해결했습니다.
이를 통해 서버에서 authorization을 진행할 때 authentication이 필요한 API 요청들은 현재 로그인 상태인 회원의 정보가 필요하므로, 로그인한 회원의 정보를 전부 사용할 수 있게 되었습니다.
